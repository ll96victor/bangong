# rules.md — 油猴脚本开发通用规则

## 一、动手写代码之前，先做环境侦察

任何脚本失败，90% 的原因是对目标页面的渲染机制做了错误假设。写代码之前必须先回答三个问题：

**目标内容在哪里渲染？**

```javascript
// 必跑的三行侦察代码
console.log('canvas:', document.querySelectorAll('canvas').length);
console.log('iframe:', document.querySelectorAll('iframe').length);
console.log('shadowRoot:', document.querySelectorAll('*').length, '个元素');
```

**目标内容是静态的还是动态加载的？**

打开 Network 面板，观察页面加载完成后是否还有持续的 XHR/Fetch 请求。有的话说明是动态渲染，脚本需要等待内容出现再执行。

**页面是 MPA 还是 SPA？**

在页面内点击导航链接，观察浏览器地址栏是否触发完整刷新。SPA 不会刷新，注入的 DOM 元素会在路由切换时消失，必须用 `MutationObserver` 重新注入。

## 二、如果`console.log` 在以下情况会静默失败或输出被淹没，则调试时永远用 `alert()` 而不是 `console.log()`

这是本次调试中最大的时间浪费来源。`console.log` 在以下情况会静默失败或输出被淹没：

- 页面有大量错误日志（本次飞书有 34 个错误 + 170 个警告）
- 控制台 Filter 过滤掉了输出
- 执行上下文不对（在扩展上下文而非页面上下文运行）
- 页面 CSP 策略限制了部分输出

`alert()` 是系统级弹窗，任何情况下都不会被屏蔽。调试阶段一律用 `alert()`，上线前再换回 `console.log()`。

```javascript
// 调试期标准写法
alert(JSON.stringify(someValue, null, 2));

// 上线后换成
console.log('[脚本名]', someValue);
```

## 三、建立明确的"放弃判断标准"，避免无效调试

每一条技术路线都应该设定一个明确的终止条件，而不是无限尝试。

| 技术路线 | 放弃条件 |
|---------|---------|
| 操作 DOM 滚动容器 | 所有候选容器 `scrollH === clientH` |
| 模拟键盘事件 | 目标元素无法获得焦点，且 `document` 级监听也无响应 |
| 操作 iframe 内容 | `contentDocument` 抛出跨域异常，或 `body.scrollHeight === 0` |
| 调用内部 API 实例 | `window` 上无相关命名空间，canvas 上无自定义属性 |
| 整体方案 | 同时满足：无可滚动容器 + 无 Canvas + 目标容器子元素为空 |

遇到放弃条件时，立即转向替代方案（原生快捷键、浏览器扩展、平台官方 API），不要继续在死路上消耗时间。

## 四、事件模拟的正确姿势

模拟用户交互是油猴脚本最常见的需求，也是最容易踩坑的地方。

**完整的鼠标事件序列，缺一不可：**

```javascript
function simulateClick(el) {
  const rect = el.getBoundingClientRect();
  const cx = rect.left + rect.width / 2;
  const cy = rect.top + rect.height / 2;
  // 必须按顺序，不能只发 click
  ['mousedown', 'mouseup', 'click'].forEach(type => {
    el.dispatchEvent(new MouseEvent(type, {
      bubbles: true, cancelable: true,
      clientX: cx, clientY: cy, button: 0,
    }));
  });
}
```

**键盘事件必须补全所有字段：**

```javascript
function simulateKey(el, key, keyCode, modifiers = {}) {
  ['keydown', 'keyup'].forEach(type => {
    el.dispatchEvent(new KeyboardEvent(type, {
      key, code: key, keyCode, which: keyCode,
      bubbles: true, cancelable: true,
      composed: true,   // 穿透 Shadow DOM，不能省
      ...modifiers,     // { ctrlKey: true } 等
    }));
  });
}
```

**事件目标的优先级选择：**

```
canvas 元素 → 具体容器元素 → document.activeElement → document → window
```

如果不确定事件监听在哪一层，就全部都发一遍，成本极低。

**焦点与键盘事件之间必须有延迟：**

```javascript
el.focus();
setTimeout(() => simulateKey(el, 'Home', 36, { ctrlKey: true }), 80);
// 80ms 是经验值，低于 50ms 经常失效
```

## 五、处理动态页面的标准模式

### 等待元素出现

```javascript
function waitFor(selector, callback, maxTries = 20, interval = 500) {
  let tries = 0;
  const timer = setInterval(() => {
    const el = document.querySelector(selector);
    if (el) { clearInterval(timer); callback(el); }
    else if (++tries >= maxTries) {
      clearInterval(timer);
      console.warn('[脚本] 等待超时:', selector);
    }
  }, interval);
}
```

### SPA 路由变化后重新注入

```javascript
function watchAndReinject(btnId, injectFn) {
  new MutationObserver(() => {
    if (!document.getElementById(btnId)) injectFn();
  }).observe(document.body, { childList: true, subtree: false });
  // subtree: false 只监听直接子节点，性能开销极小
}
```

### 初始化延迟的经验值

```javascript
// 普通网页：0ms（DOMContentLoaded 后直接执行）
// React/Vue SPA：500 ~ 1000ms
// 腾讯文档类重型应用：2000ms
// 飞书类重型应用：3000ms
// 原则：宁可等长一点，不要因为框架未初始化就报错
setTimeout(init, 2000);
```

## 六、`@grant` 权限的正确使用

```javascript
// 需要访问页面原始 window 对象时（绕过 Tampermonkey 沙箱）
// @grant unsafeWindow
const win = (typeof unsafeWindow !== 'undefined') ? unsafeWindow : window;

// 需要跨域请求时
// @grant GM_xmlhttpRequest

// 需要持久化存储时（比 localStorage 更安全，不会被页面清除）
// @grant GM_setValue
// @grant GM_getValue

// 什么都不需要时，显式声明，避免 Tampermonkey 做不必要的沙箱隔离
// @grant none
```

## 七、脚本上线前的自检清单

在提交或分享脚本之前，逐条确认：

- `@match` 规则是否覆盖了目标页面的所有 URL 变体（含子域名 `*` 通配）
- `@run-at` 是否设置正确（`document-idle` 适合大多数 SPA，`document-end` 适合静态页面）
- 是否有防重复注入的判断（`if (document.getElementById('my-btn')) return`）
- 是否有 SPA 路由变化的重新注入逻辑
- 所有 `alert()` 调试代码是否已替换为 `console.log()`
- `console.log` 是否加了脚本名前缀（方便在日志海中快速定位）
- `MutationObserver` 是否在不需要时断开（`observer.disconnect()`）避免内存泄漏
- 是否在 `try/catch` 中包裹了可能失败的操作，避免报错影响页面正常使用
- 是否有必要的异常处理机制（如 `catch (err) { console.error('[脚本] 错误:', err); }`）
- 是否有必要的性能优化（如防抖、节流、异步加载等）
- 是否有必要的用户提示（如确认弹窗、提示信息等）
- 是否有必要的日志记录（如 `console.log` 调试信息、`console.error` 错误信息等）
