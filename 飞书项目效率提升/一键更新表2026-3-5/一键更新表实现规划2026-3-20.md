## 4. 关键技术决策：回归远程调试模式

### 4.1 v2.x 方案的问题（已废弃）

v2.x 使用 `--user-data-dir` 参数复用/新建 Chrome User Data 目录。
此方案的核心缺陷：
1. **需要关闭 Chrome**：脚本独占 User Data 目录，用户必须先关闭所有 Chrome 窗口；
2. **在脚本目录生成约 700MB 数据**：Chrome 会把 ML 模型（`.tflite`、`.pb`）、缓存等大量文件写入指定的 User Data 目录，导致 `output/` 下积累了 400+ 个非预期文件，严重影响 git 同步；
3. **体验差**：不符合用户"Chrome 一直开着"的日常工作习惯。

### 4.2 v3.0 方案：远程调试模式（最开始的副本方案）

以 `one_click_update - 最开始的副本.py`（v1.5）为基础回归：
- 用户通过 `.bat` 以 `--remote-debugging-port=9222` 启动 Chrome（或用已有的调试 Chrome）；
- 脚本通过 `debuggerAddress=127.0.0.1:9222` 连接，在用户现有 Chrome 里开新标签页处理；
- **不创建任何新 User Data 目录**，不写入任何大文件到脚本目录；
- Chrome 一直保持开启，用户可以正常使用。

**调试端口 Chrome 的 User Data 存放位置**：
- `.bat` 启动时使用系统临时目录（`%TEMP%\chrome_feishu_debug`）作为 `--user-data-dir`；
- 该目录在系统盘 `Temp` 下，不影响 git 仓库；
- 用户首次启动需要在该 Chrome 窗口登录飞书（只需一次，之后 User Data 保留登录态）。

### 4.3 两个方案对比

| 维度 | v2.x（废弃）| v3.0（当前）|
|------|------------|------------|
| 是否需要关闭 Chrome | 是 | 否 |
| 脚本目录大小影响 | 约 +700MB | 无 |
| git 同步友好 | 否 | 是 |
| 登录方式 | 每次可能需要重新登录 | 首次登录后 User Data 保留 |
| 方案来源 | v2.x 新增 | 回归 v1.5 原始方案 |

## 5. 最终用户功能

### 5.1 输入阶段
用户可通过以下方式提供链接：
1. 输入 txt 文件路径；
2. 控制台直接粘贴多行链接（空行结束）；
3. 剪贴板自动读取。

### 5.2 执行阶段
1. `.bat` 检测 9222 端口是否已有调试 Chrome，没有则自动启动一个（User Data 在 `%TEMP%\chrome_feishu_debug`）；
2. 脚本连接到调试 Chrome；
3. 检测登录状态，未登录则提示用户在浏览器登录后按回车继续；
4. 顺序处理链接（每个开新标签页 → 提取 → 关闭标签页）；
5. 提取重试（最多 10 秒等待 + 结果校验重试）；
6. 支持断点续传（只跳过上次成功的链接）；
7. 完成后导出 CSV/Excel，打印汇总。

### 5.3 输出阶段
1. `output/项目信息_{时间戳}.csv`
2. `output/项目信息_{时间戳}.xlsx`
3. `output/one_click_update_{时间戳}.log`

### 5.4 输出字段（顺序固定）
`项目名称 | 状态 | 项目名称2 | 空 | 日期 | 来源 | aihelp链接 | 处理信息`
