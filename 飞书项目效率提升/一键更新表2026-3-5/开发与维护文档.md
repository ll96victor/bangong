## 2. 当前真实架构

本工具基于 **Selenium + 远程调试模式 Chrome**，核心机制：

1. 用户通过 `.bat` 以 `--remote-debugging-port=9222` 启动 Chrome
2. 脚本通过 `debuggerAddress=127.0.0.1:9222` 连接到已有 Chrome
3. 在用户现有 Chrome 里开新标签页处理，完成后关闭标签页
4. Chrome 全程保持开启，脚本结束后可直接继续使用

**关键优势**：
- 不创建任何新 User Data 目录（User Data 由 `.bat` 统一放到 `%TEMP%\chrome_feishu_debug`）
- 不影响 git 同步（脚本目录下不生成大体积文件）
- 用户无需关闭 Chrome，体验友好

整体流程：

```
双击运行.bat
  ├─ 检测 9222 端口是否已有调试 Chrome
  │   ├─ 没有 → 以调试模式启动新 Chrome（User Data 在 %TEMP%\chrome_feishu_debug）
  │   └─ 已有 → 直接连接
  └─ python one_click_update.py
       ├── prompt_links() → 用户选择链接输入方式（剪贴板 / 文件 / 控制台粘贴）
       ├── connect_to_chrome() → 连接到调试端口 9222 的 Chrome
       ├── check_login() → 检测登录状态
       ├── load_progress() → 断点续传（只跳过上次成功的链接）
       ├── 循环处理每条链接
       │   └─ extract_project_info() → 等待加载 → JS 注入提取 → 重试
       ├── save_results() → 导出 CSV / Excel
       ├── clear_progress() → 清理断点续传文件
       └── print_summary() → 汇总报告
```

## 3. 核心函数说明

### 3.1 Chrome 连接与登录检测

| 函数 | 说明 |
|------|------|
| `connect_to_chrome()` | 通过 `debuggerAddress=127.0.0.1:9222` 连接到调试模式的 Chrome |
| `check_login()` | 访问飞书首页检测登录状态，未登录则暂停等用户操作 |

**关键机制**：
- 使用 `options.add_experimental_option("debuggerAddress", "127.0.0.1:9222")` 连接已有 Chrome
- 不调用 `driver.quit()`，避免关闭用户的 Chrome

### 3.2 链接输入

| 函数 | 说明 |
|------|------|
| `prompt_links()` | 提供三种输入方式：剪贴板 / txt 文件路径 / 控制台多行粘贴（输入 p 进入） |
| `_parse_links()` | 解析文本，过滤出含 `project.feishu.cn` 的链接，去重 |
| `_read_clipboard()` | 用 tkinter 读取系统剪贴板，不依赖第三方库 |

### 3.3 页面等待与信息提取

| 函数 | 说明 |
|------|------|
| `extract_project_info()` | 等待页面加载后提取信息，带重试机制 |
| `_extract_once()` | 执行一次提取，返回原始字段字典 |

**提取脚本（JavaScript）核心逻辑**：
- `_GET_PROCESS_INFO_JS`：查找"处理信息"标签 → 取其 nextElementSibling / parentElement.nextElementSibling → 回退到 body 文本逐行查找
- `_GET_AIHELP_LINK_JS`：优先找"原单链接"容器内的 `<a>` → 页面内任意 aihelp 链接 → 正则匹配文本 → MCGG 特殊：提取 Ticket ID
- 过滤干扰词（`INTERFERENCE_WORDS`）和无效值（"待填"）

### 3.4 断点续传

| 函数 | 说明 |
|------|------|
| `load_progress()` | **只恢复上次成功的条目**，失败项不跳过（下次重试） |
| `save_progress()` | 每处理完一条链接后立即保存，防止中断丢失 |
| `clear_progress()` | 全部完成后删除进度文件 |

进度文件路径：`output/one_click_update_progress.json`

文件格式：
```json
{
  "updated_at": "2026-03-20T14:30:22",
  "results": [
    {
      "项目名称": "...", "_source_url": "...", "_success": true, "_error": ""
    }
  ]
}
```

### 3.5 导出结果

| 函数 | 说明 |
|------|------|
| `save_results()` | 生成 `项目信息_{时间戳}.csv` 和 `.xlsx`，只包含 `OUTPUT_COLUMNS` 中的列 |
| `print_summary()` | 输出总数 / 成功 / 失败 / 运行时长 / 失败链接详情 / 文件路径 |

输出字段（`OUTPUT_COLUMNS`，顺序固定）：
```
项目名称 | 状态 | 项目名称2 | 空 | 日期 | 来源 | aihelp链接 | 处理信息
```

固定值：`状态` = `定位中未修复`，`来源` = `BugGarage`，`日期` = 当天（`YYYY/MM/DD`）。

## 4. 配置项

所有配置项集中在 `CONFIG` 字典中：

```python
CONFIG = {
    "page_timeout": 30,        # 页面加载超时（秒）
    "wait_timeout": 12,        # 页面加载最多等待秒数
    "extract_retries": 3,      # 提取内容重试次数
    "extract_retry_interval": 1.5,  # 重试间隔（秒）
    "batch_size": 10,          # 每批处理数量
    "batch_delay": 2,          # 批次间延迟（秒）
}
```

## 5. 排错顺序

遇到问题时建议按以下顺序排查：

1. **查看 `output/` 下的 `.log` 文件** → 定位具体报错位置
2. **Chrome 连接失败** → 确认是否通过 `.bat` 启动，或手动以调试模式启动 Chrome
3. **登录问题** → 在调试 Chrome 窗口中完成飞书登录
4. **提取结果为空** → 用手动打开该链接验证页面是否正常加载，或查看 `extract_project_info()` 的重试日志
5. **页面 DOM 结构变化** → 在 `_GET_PROCESS_INFO_JS` 或 `_GET_AIHELP_LINK_JS` 中追加新的查找策略

## 6. 后续修改注意事项

- **不要引入新的大型依赖**，当前只依赖 selenium / pandas / openpyxl，维持最小依赖。
- **修改提取逻辑时**，优先在 JavaScript 字符串（`_GET_PROCESS_INFO_JS`、`_GET_AIHELP_LINK_JS`）中修改，因为提取在浏览器内执行。
- **修改 `OUTPUT_COLUMNS`** 时，`extract_project_info()` 中的 `result_template` 需要同步更新。
- **增加配置项**时，统一放在 `CONFIG` 字典中，不要在函数内直接写魔法数字。
- **改动输入方式**时，`prompt_links()` 和 `一键更新表使用说明.md` 必须同步修改。
- **改动断点续传逻辑**时，`load_progress()` 和 `一键更新表使用说明.md` 中"中断后如何续跑"章节必须同步修改。
