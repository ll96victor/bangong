## 3. 脚本定位与运行范围

这是一个多模块合并油猴脚本，核心作用是提升 AIHelp 工单处理效率，并补充飞书搜索和 GM 工具自动化。

### 3.1 主要能力

- 普通工单：复制内部描述、自动识别区服、自动翻译并处理标题
- MCGG 工单：复制描述、自动处理 MCGG 标题
- Task 信息：提取客服信息并复制链接
- 自动回复：提交后自动点击“内部回复”并填充固定文本
- 飞书联动：从工单中提取 `Ticket ID`，到飞书项目页自动搜索
- 清除头像：跨页打开 GM 工具，自动执行清除头像冷却时间
- 内部回复快捷入口：从状态栏图标或展开面板直接点击

### 3.2 运行页面

脚本当前通过 `@match` 覆盖：

- `https://ml-panel.aihelp.net/*`
- `https://ml.aihelp.net/*`
- `https://aihelp.net.cn/*`
- `https://aihelp.net/*`
- `https://project.feishu.cn/ml/workObjectView/onlineissue/Cot68m5vg`
- `https://project.feishu.cn/ml/workObjectView/onlineissue/Cot68m5vg?*`
- `https://gm.moba.youngjoygame.com:8090/*`

同时通过 `@exclude` 排除：

- `*://*/dashboard/#/newpage-ticket*`

脚本启动后还有一层运行时判断：

- URL 包含 `ticket` 时直接退出
- AIHelp 主逻辑只在 `task?orderId` 或 `tasks?searchType` 页面运行
- 飞书执行器只在固定飞书目标页运行
- GM 模块只在 GM 工具域名运行

## 4. 模块总览

脚本采用“多个独立 IIFE 模块 + 少量共享工具”的组织方式。

### 4.1 公共层

公共层主要包含：

- 页面判断函数
- 状态栏 `StatusbarUI`
- 共享工具 `SharedUtils`
- MCGG 判断配置 `MCGG_CONFIG`

### 4.2 模块列表

#### 模块 A：普通工单助手

主要职责：

- 提取“内部描述”
- 根据 `ServerID` 判定全服/测服
- 自动处理“任务标题”
- 自动处理“关联第三方”中的 `发现迭代`、`渠道`
- 提供普通工单复制功能

当前重要行为：

- 标题无冒号也会进入翻译流程
- 中文标题会保留交互面板
- 先处理标题，再处理 `发现迭代`，最后处理 `渠道`
- 关联第三方使用 `.tabDetail-item.thirdLink` 内行标题定位；展开 `el-select` 时优先匹配 **`.custom-down-select-search`** 弹出层，依次尝试 **input 点击 / 箭头 / 合成事件**；`fillDropdownSearch` 在自动链路中带 **`preferThirdLink`**（与手动焦点兜底共用 `SharedUtils`）
- 「渠道」若界面当前值已与 ServerID 推断的全服/测服一致则跳过自动选择
- 自动处理前约 **400ms** 延迟，便于「关联第三方」区域渲染完成
- 如果自动填充失败，仍保留焦点监听作为兜底

#### 模块 B：MCGG 工单助手

主要职责：

- 提取“描述”字段
- 根据 `ServerID` 判定标题前缀
- 自动处理 MCGG 标题
- 在标题处理完成后，同样自动处理「关联第三方」中的 `发现迭代`、`渠道`（逻辑与模块 A 对齐）
- 提供 MCGG 描述复制功能

#### 模块 C：Task 客服信息提取

主要职责：

- 提取工单链接
- 提取客服前缀与名称
- 复制为 `链接 + @客服名`

#### 模块 D：提交后自动回复

主要职责：

- 监听工单页面“确认/确定”点击
- 排除批量处理和批量筛选场景
- 自动打开“内部回复”
- 自动填充固定文本 `already submitted`
- 自动点击回复按钮

#### 模块 E-AIHelp 端：飞书搜索触发器

主要职责：

- 监听工单切换
- 提取 `Ticket ID`
- 将搜索任务写入 GM 存储（含 `ticketId`、`requestTs`、**`searchRequestId`**，便于与飞书端控制台日志对照）
- 唤起或复用飞书目标页
- 轮询飞书端结果并展示在日志面板

#### 模块 E-飞书端：飞书搜索执行器

主要职责：

- 仅在指定飞书目标页运行
- 轮询待搜索任务
- 点击“查找”
- 输入 `Ticket ID`
- 点击“过滤”
- 判断结果并回写 GM 存储

#### 模块 F-AIHelp 端：清除头像

主要职责：

- 从工单中提取 `UID` 和 `ServerID`
- 写入跨域任务
- 打开 GM 工具页面
- 轮询读取 GM 端日志

#### 模块 F-GM 端：清除头像执行器

主要职责：

- 读取 AIHelp 写入的任务
- 在 GM 工具页自动选择区服、输入 UID、执行操作
- 实时写日志到 GM 存储

#### 模块 G：内部回复快捷入口

主要职责：

- 在状态栏和展开面板中提供“内部回复”快捷按钮
- **未打开**内部回复对话框时：自动点击工单工具栏上的「内部回复」或「内部备注」
- **已打开**标题为「内部回复」的对话框（如 `.cusHeader` 文案为内部回复）时：自动点击对话框底部 **「回复」**（与再次打开工具栏按钮区分）

## 6. 关键共享工具说明

`SharedUtils` 是当前脚本最重要的公共能力层，很多功能都依赖它。

### 6.1 当前关键方法

- `getCurrentTicketID()`
  当前已优化为：
  - 先从 URL 提取 `orderId`
  - 再扫描局部容器
  - 最后回退到 `document.body`

- `findTitleInputRobust()`
  标题输入框的鲁棒查找逻辑

- `extractTextByFormLabels()`
  基于字段名的 outerHTML 提取逻辑

- `extractContentWithImages()`
  文本提取时保留图片 URL

- `copyText()`
  复制时先走 `navigator.clipboard`
  失败后回退 `GM_setClipboard`

- `waitForDropdownSearchInput()`
  查找当前展开下拉框中的搜索框（可选 `preferThirdLink`，优先 `.custom-down-select-search`）

- `fillDropdownSearch()`
  向下拉搜索框输入文本，并尝试自动点选匹配项（可选 `preferThirdLink`）

- `selectDropdownOption()`
  不再只是“输入关键词”，而是尽量真正选中下拉项

- `findThirdLinkFieldRow` / `findThirdLinkFieldInput` / `getThirdLinkFieldDisplayValue`
  关联第三方区域内按标签定位行与输入框

- `openThirdLinkElSelectDropdown(labelTexts, fieldName, logFn)`
  打开关联第三方 `el-select` 下拉（input / caret / 合成事件多重尝试）

- `channelDisplayMatchesDesired(displayValue, desiredChannel)`
  判断渠道当前展示是否与目标全服/测服一致，用于跳过重复操作

### 8.4 最近几轮重点结论（v6.8.14–v6.8.18）：发现迭代稳定性回收

这几轮对话里，和维护最相关的结论只有一条：

- “发现迭代”出错时，先不要怀疑 `ServerID` 推导或目标值计算

本次已确认过的实际根因：

- 日志已经证明模块 A 算出的目标值是对的，例如测试服会得到 `2.1.66`
- 用户看到的旧值 `2.1.60`，更可能是页面原本残留值，没有被成功改掉
- `v6.8.17` 把“发现迭代”回退到了 `fillDropdownSearchLegacy -> selectDropdownOptionLegacy`
- 这条旧链路只看较短时间内的候选命中，不再做更稳的候选等待和最终字段值确认
- 一旦候选刷新偏慢，或者候选实际显示为 `2.1.66(非C#)` 这类扩展文本，就容易出现“没真正选中，页面继续保留旧值”

`v6.8.18` 的实际修正：

- 模块 A 和 MCGG 的“发现迭代”都改回 `fillDropdownSearch(..., { preferThirdLink: true })`
- 搜索后统一补 `confirmThirdLinkFieldApplied(...)`
- 确认等待时间对“发现迭代”放宽到 `Math.max(CONFIG.thirdLinkConfirmTimeout, 2600)`
- 成功判定兼容 `2.1.66(...)` 这类扩展文本，不再强制全等
- 失败日志补“当前字段值”，便于区分“候选没刷出来”和“页面仍停留旧值”

后续维护时请记住：

- “发现迭代”不要再默认退回 legacy 专用链路
- 这个字段的稳定规则应当是“正常搜索 + 最终字段值确认”
- 若未来 AIHelp 页面升级导致下拉结构变化，优先排查 `.custom-down-select-search`、候选项 DOM 和字段展示值确认，而不是先改版本计算逻辑

### 8.5 v6.8.19：发现迭代误点旧值与工单切换残留

本轮用户需求：

- 模块 A 和 MCGG 的“发现迭代”偶尔会先点到旧版本，例如目标应为 `2.1.66`，第一次却点到旧候选 `2.1.60(非C#)`
- 个别场景里，“发现迭代”输入框甚至会出现 `测服 / 全服` 这类本应属于“渠道”的值，怀疑是上一单异步动作残留到当前工单

原有行为：

- “发现迭代”自动填充会复用通用下拉搜索逻辑
- 该逻辑在候选列表只剩 1 项时，允许直接走“唯一候选”兜底点击
- 工单切换时虽然会重置状态，但旧工单已经启动的异步自动填充链路没有额外校验“这还是不是当前工单”

本次修改：

- 模块 A / MCGG 的“发现迭代”去掉“唯一候选”兜底，只保留正常候选匹配与最终字段值确认
- 模块 A / MCGG 的标题处理、发现迭代、渠道、创建人、功能模块自动填充都补了工单上下文校验
- 一旦工单 ID 已切换，旧工单尚未跑完的自动处理链路会直接终止，不再继续操作当前页面

使用方法：

- 继续按原来方式处理工单即可，无需新增按钮或配置
- 若手动点击“发现迭代”输入框，焦点兜底仍然会执行，只是不会再因为“唯一候选”而误点旧版本

注意事项：

- 这次收紧的是“发现迭代”的唯一候选兜底，不影响“渠道 / 创建人 / 功能模块”的正常自动选择
- 如果后续又出现“发现迭代为空”，优先看日志里是否是“候选尚未刷新”或页面根本没返回候选，而不是先怀疑 `ServerID`
- 如果后续还出现“测服 / 全服”串到错误字段，优先排查页面当前 DOM 是否存在复用输入框或焦点落错行

### 8.6 v6.8.20：发现迭代串入上一单搜索词

本轮新增用户反馈：

- 上一单如果是 MCGG，下一单普通工单的“发现迭代”里会出现 `模式独立包`
- 上一单如果是普通工单测服 / 全服，下一单普通工单的“发现迭代”里会出现上一单的 `2.1.66` 或 `2.1.60`

这说明问题不只是“目标值计算”或“旧异步链路”：

- 更像是上一单遗留的关联第三方下拉弹层还处于打开状态
- 下一单再次处理“发现迭代”时，脚本误复用了旧弹层里的搜索框
- 于是上一单的搜索词被用户看到，形成“模式独立包 / 2.1.66 / 2.1.60 串到新工单”的现象

本次修改：

- `SharedUtils` 新增“关闭并清空可见关联第三方下拉弹层”的通用处理
- 每次打开新的关联第三方字段前，先清空旧弹层搜索框并主动关闭旧弹层
- 普通工单 / MCGG 在工单切换 `resetState()` 时，也会额外触发一次旧弹层清理

使用方法：

- 仍按原方式切换工单和处理工单即可，无需新增配置
- 如果切单很快，脚本会先清旧弹层，再开当前字段的新弹层

注意事项：

- 这次修的是“旧下拉弹层 / 旧搜索词残留”，不是删除 `ServerID` 推导或版本回填逻辑
- 如果后续仍看到串词，优先确认页面是否真的同时存在多个可见 `.custom-down-select-search` 弹层

## 9. 当前已知问题与风险

### 9.1 元信息版本不一致

**状态：已处理（v6.8.5 起）**：脚本头 `@name` 与 `@version` 已统一为同一版本号（当前 **6.8.20**）。

**仍须注意**：仓库内文件名仍为 `工单助手与Task客服信息提取合并版 6.8.0-6.8.0.user.js`，与版本号数字不同，交接时以脚本头为准。

### 9.2 飞书搜索仍属于高波动 DOM 自动化

虽然本轮增加了：

- 心跳复用
- 目标页打开锁
- 更长超时
- 搜索中不消费新任务

但飞书页面本质仍是重型 SPA，DOM 结构或交互机制一变，搜索流程仍可能失效。

重点风险点：

- “查找”按钮结构变化
- 搜索框 placeholder 变化
- “过滤”复选框结构变化
- 结果判断依赖 `X/X` 计数和空态文本，不是官方 API

### 9.3 下拉框自动选择对页面实现有依赖

模块 A 和模块 B 的下拉自动处理目前依赖：

- `.el-select-dropdown`
- `.el-select-dropdown__item`
- 展开后存在搜索输入框

如果 AIHelp 后台升级了 Element UI 版本或组件实现，自动填充可能先失效。

其中“发现迭代”字段还要额外注意：

- 这个字段在真实页面里容易出现候选刷新慢、候选文本带后缀、点击成功但字段回填稍晚
- 因此它不能只依赖“候选列表里有没有马上匹配到”
- 当前稳定做法是：先走 `fillDropdownSearch(..., { preferThirdLink: true })`，再用 `confirmThirdLinkFieldApplied(...)` 看最终字段展示值是否生效
- 如果后续又看到日志里出现连续“未找到可点击的下拉选项”并伴随页面保留旧值，优先检查是否有人又把“发现迭代”切回了 legacy 链路

### 9.4 `currentUrl` 存在多处局部重名

脚本中多个 IIFE 内都定义了自己的 `currentUrl`。

当前能正常运行，但维护时要注意：

- 不同模块引用的是自己的局部变量，不一定是同一个
- 拆代码或移动代码块时容易引入作用域错误

### 9.5 多模块仍然是多套轮询

目前多个模块仍各自维护 `setInterval`。

这在可读性和维护性上仍然偏重，后续如果继续中高风险重构，可以考虑统一“工单切换总线”。

## 12. 开发过程经验总结

这部分是本轮维护最值得保留给后续接手者的内容。

### 12.1 DOM 自动化不要只相信“结构稳定”

实际情况是：

- 相同页面在不同加载阶段，DOM 差异很大
- 同一字段可能有多种渲染方式
- 只靠一个选择器很容易偶发失效

有效做法是多层兜底：

- outerHTML 优先
- 精确 DOM 提取次之
- 文本扫描最后兜底

### 12.2 复制功能不能只依赖一个剪贴板 API

在油猴环境里，`navigator.clipboard` 有时受页面权限或浏览器策略影响。

经验结论：

- 先用 `navigator.clipboard`
- 失败后立即回退 `GM_setClipboard`

这比单一路径稳定得多。

### 12.3 下拉框自动化不能停在“输入关键字”

本轮一个明确教训是：

- 把字填进搜索框，不代表值真的选中了

所以后来补了：

- 展开下拉框
- 输入关键字
- 再尝试真正点击匹配项

最近几轮又补了一层更重要的经验：

- 对“发现迭代”这种字段，`点击候选成功` 也不等于最终值一定已经生效
- 真正稳定的判定应该落在“字段最终展示值”上
- 只要页面最终显示的是目标版本，哪怕候选链路日志不够漂亮，也应该按成功处理
- 相反，如果页面仍保留旧值，就不能因为前面搜索链路跑过了就误判成功

如果后续还有类似字段，优先复用这一思路。

### 12.4 飞书这种重型 SPA，单靠心跳还不够

之前只做“心跳复用”还不够稳，原因是：

- 目标页刚打开但脚本还没开始写心跳时
- AIHelp 端可能又触发了一次打开

后来补上的关键是“打开锁”：

- 页面还没活过来之前，先标记“已经在打开中”
- 减少重复开多个目标页

这类跨页自动化，通常需要：

- 心跳
- 打开锁
- 请求时间戳
- 结果时间戳

四者一起用才更稳。

### 12.5 生产脚本重构要分批做

这份脚本已经是多模块长期迭代结果。

经验上不要一次动太多层：

- 第一批先做低风险清理
- 第二批再做中风险优化
- 每批都要能单独回归

这比“想一次性整理漂亮”更实际。

### 12.6 注释和文档非常重要

这类脚本的问题不是“功能少”，而是：

- 页面多
- 状态多
- 兜底多
- 时间久了很容易忘记为什么这么写

所以后续维护时，建议遵守：

- 新增一个跨页状态，就写明用途
- 新增一个兜底分支，就写明触发原因
- 改动一轮中风险逻辑，就同步更新维护文档

## 15. 交接结论

当前脚本并不是“全新重构完成态”，而是“已经可维护、可继续演进”的阶段。

它的维护策略应当是：

- 保持模块边界清晰
- 在高波动 DOM 场景下保留多层兜底
- 用小步快跑替代一次性大重构
- 每一轮改动后补文档，减少未来维护成本

补充一句这次交接里最关键的判断：

- 如果用户反馈“发现迭代又变回旧值”，先查页面最终字段值和下拉链路，不要先怀疑版本识别逻辑
- 当前 `v6.8.20` 的主线结论是：目标值计算基本没错，问题更常出在旧下拉弹层没关干净、旧搜索词残留和旧工单异步链路残留

如果后续继续迭代，这份文档应当随版本一起更新，而不是只写一次。
